# Lightwave Git — Workflow Discipline

This skill exists because previous sessions cost full days on git mess. The Lightwave workspace is a multi-repo setup with `lightwave-media/root-folder` as the umbrella, plus `lightwave-platform`, `lightwave-cli`, `lightwave-core`, `lightwave-ui` as siblings. Each has its own pre-commit hooks, branch-naming gates, and CI. **Follow the rules. The cost of "I'll just" is enormous.**

---

## Hard rules (non-negotiable)

### Branch hygiene

1. **Always start a new branch from `origin/main`.** Fast-forward local main first, then branch.
   ```bash
   git fetch origin && git checkout main && git pull --ff-only && git checkout -b fix/your-branch main
   ```
2. **One logical change per branch.** No mixing concerns. If you find yourself stashing across branches, you have too many concerns mixed in.
3. **Branch naming convention is enforced by pre-push hook.** Use `fix/`, `feature/`, `chore/`, `style/`, `docs/`, `ci/`, `refactor/`. Include LIGA-N if applicable.
4. **Delete branches when their PR merges.** Squash-merges leave branches that `git branch -d` won't recognize as merged — use `-D` after verifying via PR. Local-only branches that were never pushed: verify content with `git log <branch> ^main --oneline` first.
5. **`backup/*` branches are technical debt.** They're a symptom of "I might lose work" anxiety — usually the work is already in stashes or commits. If you create one, file an issue with a deadline; don't let it sit.

### Stash hygiene

1. **Stash is for <30 minutes of context-switching, not days.** Pop or drop within the working session. Otherwise the stash becomes stale, the parent branch evolves, and applying it produces garbage.
2. **Auto-save stashes from `claude-session-start` are ephemeral.** Drop them aggressively. They snapshot working tree at session boundaries and are immediately superseded. They should not accumulate past a single day.
3. **Never `git stash pop` blind.** Always `git stash show stash@{N} --stat` first to see the file list. If files in the stash are now deleted on main (e.g. removed in a recent PR), the pop will produce conflicts that take 10x more time than just discarding.
4. **Name your manual stashes.** `git stash push -m "WIP: <what & why>"`. "WIP" alone is useless three days later.

### Working tree hygiene

1. **Never switch branches with a dirty working tree** unless the changes apply cleanly to the target. Carrying staged/unstaged changes across branches via `git checkout` is how you lose track of what's intended for which branch.
2. **Commit or stash before switching.** Even a one-file edit. Past you should not have to reverse-engineer "why is this random hooks/stop.ts edit on the formatting branch?"
3. **Lock files (`uv.lock`, `pnpm-lock.yaml`) are auto-regenerated by direnv hooks.** They are in `.prettierignore`. If they show as modified after a `git checkout` or `git pull`, that's a sync side-effect — `git restore` them unless you intentionally bumped a dep.

### Worktrees

1. **Worktrees do NOT work for cross-repo Makefile invocations.** Lightwave's `lw make platform <target>` runs make in a sibling repo whose Makefile uses `git rev-parse --show-toplevel` and resolves WORKSPACE_ROOT to the **canonical** parent path, not your worktree. Worktree-based commits that trigger such hooks (e.g. `compose-verify`) will fail.
2. **Use worktrees only when the change is pure file-level content** (docs, hooks, SKILL.md, scripts that don't shell out to siblings). For anything that touches generated files, SST regen, or sibling-repo CLI invocations → work in the canonical checkout.
3. **Always clean up worktrees:** `git worktree remove <path>`. Don't `rm -rf` them — leaves dangling refs. `git worktree prune` periodically.

### Pre-commit & pre-push

1. **Hooks may modify your staged files.** Prettier, ruff, and ruff-format reformat in place during commit. The commit FAILS so you can re-stage. **Always re-stage and retry**, do NOT `--no-verify`.
2. **`--no-verify` is forbidden** unless the user explicitly authorizes it for a specific commit. The hook is failing because of a real problem; bypassing it ships the problem.
3. **Prettier oscillation is real.** If prettier reformats a file differently on each pass (indentation grows by N each time, no convergence), the file is in a bug state. **Add it to `.prettierignore`** rather than playing whack-a-mole. Generated files (lock files, compose YAML) and JSX-in-markdown templates (skill files with embedded React snippets) are the usual offenders.
4. **Branch-naming hook rejects on push.** If your branch is `chore/foo`, rename to `fix/foo` (or other allowed prefix). Don't fight the regex.

### Commits

1. **Conventional commits required by hook:** `type(scope): description`. Types: `feat`, `fix`, `chore`, `docs`, `style`, `refactor`, `test`, `ci`, `perf`, `build`, `revert`.
2. **One concern per commit.** Mixed commits poison `git revert` and `git bisect`.
3. **Commit messages explain WHY.** "Fix bug" is useless. Explain the failure mode and how this addresses it.
4. **`Co-Authored-By: Claude Opus 4.7` is fine.** `Co-Authored-By: Claude Haiku` indicates the stop hook was bypassed — investigate before merging.

### Conflict resolution

1. **Conflicts that span 1000+ lines mean the cherry-pick/rebase target is too stale.** Don't grind through them — abort, find the specific behavioral change you need, and re-implement against current main as a fresh commit.
2. **`--ours` / `--theirs` is a smell.** Means you're picking a side without understanding what you're losing. Acceptable only when the file is in `.prettierignore` (so the side you pick won't auto-revert next commit) or the conflict is in trivial whitespace.
3. **`git cherry-pick --abort` is cheap.** If something feels off mid-cherry-pick, abort and reassess. The aborted state is a no-op.

### Recovery

1. **Lost a commit?** `git reflog` shows every HEAD move for 90 days. The SHA is recoverable: `git checkout <sha>` or `git cherry-pick <sha>`.
2. **`git fsck --unreachable --no-reflogs`** finds genuinely-orphaned objects.
3. **Don't `git reset --hard`** unless you've confirmed via `git status` and `git stash list` that nothing is going away. The harness blocks this for a reason.

## Decision flowchart

```
Need to make a change on main?
├── Already merged this concern? → branch off main, work, PR
├── Iterating on a PR I authored? → checkout PR branch, push more commits
└── Recovering lost work? → git reflog, then cherry-pick

Working tree has uncommitted changes?
├── Same concern as current branch? → commit
├── Different concern? → stash (with -m), switch, work, stash pop
└── Auto-save noise? → drop the stash; the parent will re-create it

Pre-commit failed?
├── Hook modified my files (prettier/ruff)? → re-stage, retry
├── Hook found a real issue (secret, type error)? → fix the issue
├── branch-naming failed? → rename branch
└── Hook itself is buggy? → file an issue, do NOT --no-verify

Want to "save" current state for later?
├── Within session, will resume same task? → stash with name, pop in <2h
├── Across sessions, real work? → commit on a branch (even WIP)
└── "Just in case"? → don't. trust git reflog.
```

## Lightwave-wide gotchas

### Squash-merge implications

All PRs across the workspace are squash-merged. After PR #N merges:
- The branch's tip SHA is **not** an ancestor of main's tip
- `git branch -d <branch>` will refuse with "not fully merged" — this is a false negative
- Verify via `gh pr view N --json mergedAt` then use `git branch -D`
- The Claude Code harness blocks `-D` from agents — user must run it

### Bash-guard

The `bash-guard.ts` hook rejects raw commands that have `lw` equivalents. If you find yourself wanting to run raw `docker compose`, `pre-commit run`, `tsc`, `make`, etc., **stop and use `lw`**. Don't try to circumvent — the hook strips quoted strings + heredoc bodies before matching, so escape attempts fail.

### Multi-repo coordination

This repo lives in a workspace alongside other Lightwave repos (`root-folder`, `lightwave-platform`, `lightwave-cli`, `lightwave-core`, `lightwave-ui`). A change here may need a coordinated change elsewhere. Before starting:
- Check if the concern crosses repo boundaries (e.g. SST schema changes affect downstream consumers)
- If yes, coordinate the PR sequence — usually `lightwave-core` (schema) → `lightwave-cli` / `lightwave-ui` / `lightwave-platform` (consumers) → `root-folder` (regenerates compose etc.)
- Do not merge a downstream PR before its upstream dependency

## Common bad states & recovery

### "I have N stashes from past sessions"

1. `git stash list > /tmp/stashes.txt`
2. Audit: separate `claude-session-start:*` (auto-saves, drop all) from manual `WIP:*` (inspect each).
3. For each manual stash, check: does it touch files that still exist on main? Is the work labelled or just "WIP"? If files are deleted or label is meaningless, drop.
4. Drop in **descending index order** to keep refs stable:
   ```bash
   for i in 28 27 26 ... 0; do git stash drop "stash@{$i}"; done
   ```

### "I have N stale local branches"

1. `git branch -vv` shows tracking state. `[origin/...: gone]` means remote was deleted (typically merged via squash).
2. For each: `git log <branch> ^main --oneline` to see unique commits.
3. Empty result + remote gone → safe to delete.
4. Non-empty + remote gone → check if commits are squash-merged into main (compare commit messages, not SHAs).
5. Delete with `-d` first, fallback to `-D` only after verifying merge state.

### "Cherry-pick is producing massive conflicts"

The target is too stale. Don't grind through. Instead:
1. `git cherry-pick --abort`
2. Identify the specific behavioral change the commit introduces (read the commit message, the diff)
3. Apply that change manually as a fresh commit on a branch off current main

### "Pre-commit keeps modifying the same file in a loop"

The file is in a prettier/formatter oscillation bug. Don't keep retrying.
1. Run prettier manually 2-3 times on the file
2. If indentation/spacing keeps growing or oscillating, the file is in a bad state
3. Add to `.prettierignore` with a comment explaining why
4. Commit the `.prettierignore` change first, then the original change

### "I switched branches and now random files show as modified"

1. Likely lock files auto-regenerated by direnv (`uv.lock`, `pnpm-lock.yaml`) or session-start hook auto-staged something.
2. `git diff <file>` to see what changed. If it's pure formatting/regen and you didn't touch it intentionally, `git restore <file>`.
3. If the change is unfamiliar AND substantive, check `git stash list` and `git reflog` — past you may have left state.

## Anti-patterns (what NOT to do)

- ❌ Branching off a stale local main without `git pull --ff-only` first
- ❌ `git stash` then never popping, accumulating dozens
- ❌ Bypassing pre-commit with `--no-verify` because "the hook is wrong"
- ❌ Hand-editing generated files to fix the symptom of an upstream issue
- ❌ Re-formatting generated files (lock files, compose.yml) and committing the diff
- ❌ Cherry-picking 7+ day-old commits without checking what's evolved on main
- ❌ Mixing concerns in one branch
- ❌ Creating worktrees for work that touches sibling-repo Makefile targets
- ❌ Letting working-tree changes follow you across `git checkout`s
- ❌ Restoring `backup/*` branches via cherry-pick instead of just deleting

## When in doubt

Ask. Two-minute conversation beats two hours of cleanup.
